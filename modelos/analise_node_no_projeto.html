Decisão de arquitetura · D-0001

Node neste projeto: o que ganhamos sem ele, o que ganharíamos com ele.

O portal roda hoje sem nenhuma ferramenta de build — só arquivos estáticos que o navegador lê direto. Este documento mapeia esse desenho com honestidade: onde é um acerto, onde cobra um preço e o que o Node abriria — aqui e em projetos futuros. Há duas camadas de leitura: técnica e para todos.

Acerto hoje

Zero toolchain. Clona, dá duplo-clique no start.bat e roda. Publicar é só git push → GitHub Pages. Nada para instalar, nada que quebre no meio.

O preço

Versão dos arquivos carimbada à mão (?v=16) e bibliotecas vindas de CDN em tempo de execução — frágil e fora do nosso controle.

A oportunidade

Cache-busting automático, testes e bundle são os ganhos de maior retorno — e dá para chegar a parte deles sem virar projeto de framework.

Como o portal roda hoje — sem Node

Vanilla JS (ES Modules) + CSS puro + Supabase. Sem framework, sem bundler, sem etapa de build. Registrado na decisão D-0001.

Sem build, o cache vira trabalho manual: a versão ?v=16 é carimbada à mão em ~19 pontos de index.html (links CSS, importmap, config.js, ds.js, app.js). Esquecer um = usuário com um pé no código velho.
Em termos simples: o sistema é só um punhado de arquivos que o navegador abre direto, como abrir um documento. Não existe uma etapa de “montagem” antes de publicar. Isso deixa tudo leve e simples — mas joga para soluções manuais (ou para serviços de fora) três tarefas que uma “montagem” faria sozinha: juntar e encolher os arquivos, avisar o navegador quando algo mudou e trazer as bibliotecas de terceiros.
Sem bundler, três responsabilidades caem fora do nosso processo: empacotamento/minificação, invalidação de cache e resolução/pin de dependências. Hoje resolvidas, respectivamente, por nada (módulos soltos), ?v=N manual e import de esm.sh em runtime.

Vantagens de adotar Node (build / ferramentas)

O que uma etapa de build resolveria — com onde se aplica no código e a quem beneficia (desenvolvimento ou experiência do usuário).

A1 Cache-busting automático (hash de conteúdo)

O build renomeia cada arquivo pelo hash do conteúdo (app.9f3c.js). Some o ?v=16 manual e o risco de servir mix de versões.

Para todos: hoje, para o sistema atualizar na tela de todos, alguém precisa lembrar de “trocar um número” em vários lugares. A montagem faria isso sozinha — o usuário sempre veria a versão mais nova, sem combinação esquisita de partes antigas e novas.

O que a escolha “sem Node” evitou

Por que essa decisão foi acertada para o contexto: ferramenta interna, time enxuto, máquina corporativa, continuidade por git.

Sem npm install, sem node_modules, sem drift de lockfile. Clona e roda.

Para todos: não há “preparar o ambiente” antes de mexer no projeto. Abre e usa.

Rotas “sem Node para fazer funcionar” que pesam

Os contornos que o projeto usa para evitar Node — e por que alguns cobram um preço real, em desenvolvimento e para o usuário.

C1 Carimbar ?v=N à mão em todo lugar

Frágil: esquecer um ponto entrega mix de arquivos novos+velhos → bug sutil e intermitente que só o usuário vê. É exatamente a tarefa que um build elimina.

Para todos: depende de uma pessoa lembrar de trocar um número em vários cantos. Esquece um e parte do sistema fica “travada na versão antiga” — sem erro visível, só comportamento estranho.

C2 Importar libs críticas de esm.sh em runtime

O app depende da CDN estar no ar e liberada no firewall. Se esm.sh cai ou é bloqueado, o Supabase (app inteiro) e o import/export Excel param. Sem pin de versão; terceiro serve o código executado.

Para todos: peças essenciais são “emprestadas” de um site de fora toda vez que alguém abre o sistema. Se esse site sair do ar ou for bloqueado pela rede da empresa, o portal pode simplesmente não abrir.

Ganhos futuros — neste app e em projetos no geral

Além do que já existe: o que uma base com Node/ferramentas abre quando o sistema cresce (ex.: o módulo Cronograma a caminho) ou em outros projetos. Nem tudo se justifica hoje — é o mapa do que estaria disponível.

Componentização e reuso de verdade

Componentes com estado e props (Web Components, Lit, Svelte) no lugar de ampliar o helper h(); menos repetição quando o Cronograma reaproveitar a grade.

Para todos: peças de tela reaproveitáveis — construir telas novas fica mais rápido e mais consistente.

TypeScript

Tipos numa base que cresce (grid + store + EY) pegam erros de contrato no editor, não em produção. Migrável aos poucos via checkJs.

Para todos: o editor avisa o programador quando duas partes do sistema “não combinam”, antes de virar bug na tela.

Testes E2E e unitários

Playwright simula o usuário (login, editar célula, sync EY); Vitest cobre undo/redo e parsing. Regressão sob controle conforme o time cresce.

Para todos: um “robô” repete os caminhos principais a cada mudança e avisa se algo importante parou de funcionar.

Acessibilidade automatizada

axe-core no CI verifica contraste, foco e ARIA a cada PR — reforça a meta WCAG AA do PRODUCT.md sem depender de inspeção manual.

Para todos: garante de forma contínua que o sistema continue legível e usável por teclado e leitores de tela.

Pré-render / SSG

Astro/Eleventy gerariam páginas estáticas otimizadas — útil se alguma área (ex.: ajuda, relatórios públicos) precisar de SEO/performance. Mantém o deploy estático.

Para todos: páginas que abrem instantâneas e aparecem bem em buscas, caso algum dia algo precise ser público.

Otimização de imagens e assets

Conversão para WebP/AVIF, sprites e compressão no build — logo, mascote Ivy e prints entregues no tamanho certo.

Para todos: imagens leves = telas que carregam mais rápido, sem perder qualidade.

Design tokens compartilhados

Tokens (cores/tipografia do DS v2) exportados de uma fonte única para vários projetos do Eduardo via build — coerência de marca entre apps.

Para todos: a mesma identidade visual, sem retrabalho, em todos os sistemas da equipe.

Catálogo de componentes (Storybook)

Galeria viva de botões, badges de status, modais — desenvolve e revisa cada estado (hover/foco/erro) isolado, alinhado ao register product.

Para todos: uma “vitrine” das peças do sistema, garantindo que botão, alerta e menu tenham sempre a mesma cara.

PWA / uso offline

Service worker gerado no build daria cache resiliente e instalação como app — mitiga justamente o risco de rede da seção anterior.

Para todos: o sistema continuaria abrindo mesmo com a internet instável, e poderia ser “instalado” como um aplicativo.

Scripts e automações próprias

Node abre a porta para automações fora do Supabase (ETL da EY, geração de relatórios, migrações) escritas no mesmo ecossistema do front.

Para todos: tarefas repetitivas (importar, gerar relatório) passam a rodar sozinhas, sem ninguém fazer no braço.

Recomendação enxuta — respeitando a D-0001

Não trocar a arquitetura nem introduzir Node sem decisão explícita. Dá para matar os dois piores problemas sem Node nenhum, e reservar o build para quando o retorno justificar.

Carimbo de versão automático

Um script (Python ou PowerShell — você já tem) reescreve o ?v=N em todos os pontos a partir de um número único. Acaba o erro humano do C1.

Se um dia abrir a porta ao Node, ordem de maior retorno: 1 · cache-busting + bundle › 2 · testes da grade › 3 · CI no push

Critério do Eduardo (“isso precisa existir agora? tem consumidor?”): o Node entra quando um consumidor real aparecer — o módulo Cronograma reaproveitando a grade, ou a necessidade concreta de testes — não antes. Até lá, os passos 1 e 2 entregam o maior ganho de robustez sem ferir nenhuma regra do projeto.

Glossário — para quem não é dev

Os termos técnicos deste documento, em linguagem simples. Clique para expandir.

Node.js ferramenta
Um programa que roda código JavaScript fora do navegador, no computador. É a base de quase todas as ferramentas modernas de “montagem” de sites. Este projeto não usa Node no dia a dia — é justamente o tema do documento.
Build / etapa de build processo
A “montagem” do site: um passo automático que junta, limpa e prepara os arquivos antes de publicar. Aqui não existe — os arquivos vão para o ar como estão.
Bundler ferramenta
O “empacotador” que faz essa montagem (ex.: Vite, esbuild, webpack). Junta dezenas de arquivos em um pacote enxuto e cuida das versões.
Minificação otimização
Encolher o código removendo espaços e nomes longos. O programa fica menor e carrega mais rápido, sem mudar o que faz.
Cache-busting conceito
O navegador guarda cópias dos arquivos para abrir rápido (cache). Para ele perceber que algo mudou, trocamos a “etiqueta” do arquivo (aqui, o ?v=16). Se a etiqueta não muda, o usuário continua vendo a versão antiga.
CDN serviço
Rede de servidores externos que entrega arquivos (bibliotecas, fontes) pela internet. Prático, mas o sistema passa a depender desse serviço estar no ar e liberado pela rede da empresa.
ES Modules técnico
O jeito moderno de o JavaScript dividir o código em vários arquivos que se importam (import). Por exigirem um endereço http://, é por isso que existe o start.bat (não dá para abrir o arquivo direto).
Lint qualidade
Um “corretor automático” que lê o código e aponta erros e descuidos antes de rodar. Reduz bugs que chegariam ao usuário.
CI/CD automação
Automação que, a cada mudança, confere o código (testes/lint) e publica sozinha se estiver tudo certo. Um “porteiro” entre o trabalho e o ar.
Vendoring prática
Trazer as bibliotecas de terceiros para dentro do projeto (versionadas no repositório) em vez de buscá-las de fora a cada uso. Remove a dependência de CDN.
Supabase backend
O serviço de nuvem que guarda os dados, faz login, sincroniza em tempo real e controla permissões. É o “cérebro” por trás da grade — independe de Node.
